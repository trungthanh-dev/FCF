# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

FCF (Fuel Consumption Forecasting) predicts ship fuel consumption (`Consumer_Total_MomentaryFuel`) at multiple forecast horizons (1, 5, 10, 20 steps) using time-series sensor data from three vessels — Poseidon, Triton, Ceto — from the FuelCast dataset (loaded from `hf://datasets/krohnedigital/FuelCast/...` via `pandas.read_parquet`). Three model families are compared: Random Forest, direct-forecast LSTM (one model per horizon), and a seq2seq encoder-decoder LSTM (one model per ship, predicts all horizons at once).

There is no requirements.txt/pyproject.toml — dependencies (pandas, numpy, scikit-learn, torch, streamlit, matplotlib, joblib) come from the Anaconda base environment. Python 3.12.

## Running

All entrypoint scripts live in `src/` and use **flat, non-package imports** (e.g. `from eda import dataset_overview`, not `from src.eda import ...`). Run them from inside `src/`, or ensure `src/` is on `PYTHONPATH`:

```bash
cd src
python main.py              # full pipeline: fetch raw data from HF hub, EDA, preprocess, save data_clean/*.parquet, run Random Forest
python main_lstm.py         # LSTM only — reuses existing data_clean/*.parquet, does NOT refetch/redo EDA
python main_seq2seq_lstm.py # seq2seq LSTM only — same data_clean/*.parquet
python diagnose_lstm.py     # checks predictions_cache/lstm/*.npz for "collapsed" (near-constant) predictions
streamlit run dashboard_app.py  # interactive what-if dashboard (Random Forest models only)
```

`main.py` must be run at least once (or `data_clean/*.parquet` must already exist) before `main_lstm.py` / `main_seq2seq_lstm.py` / `dashboard_app.py` will work.

Files under `src/models/*.py` append their parent directory to `sys.path` at import time (`sys.path.append(os.path.dirname(os.path.dirname(os.path.abspath(__file__))))`) specifically so `from config import RANDOM_STATE` resolves regardless of the caller's working directory — preserve this pattern if adding new model modules.

There is no test suite and no linter configured in this repo.

## Pipeline architecture

The pipeline is a strict linear chain; each stage's output feeds the next:

1. **Load** (`main.py`) — raw parquet per ship from the HF hub.
2. **EDA** (`eda.py`) — dataset/target overviews, writes `eda_output/eda_report.txt` and per-ship stats/plots. Side-effecting only, doesn't transform data.
3. **Feature policy** (`feature_config.py`) — the single source of truth for what a "feature" is. `LEAKAGE_KEYWORDS` (ShaftPower, ShaftTorque, RotationSpeed, MomentaryFuel) identify propulsion-related columns that would leak the target; `get_features()` is the canonical feature list used everywhere downstream (never hand-roll a feature list elsewhere).
4. **Preprocess** (`preprocessing.py`, `preprocess()`) — applied per-ship, in this fixed order: remove leakage columns → remove all-null columns → remove non-predictive columns (e.g. FuelType) → `ffill`/`bfill` → add relative-bearing sin/cos direction features → add target lag/rolling/slope history features → re-run `ffill`/`bfill` (the lag/rolling step introduces fresh NaNs at series start). Output is cached to `data_clean/{ship}_clean.parquet`.
   - **Target history features are look-back only, by construction**: `add_target_history_features()` always calls `.shift()` before `.rolling()`, so a rolling feature at row *i* only ever sees rows `[i-window, i-1]`. When adding any new feature derived from the target, preserve this shift-before-rolling order — this is the project's core anti-leakage invariant, called out explicitly in both `feature_config.py` and `preprocessing.py` comments.
5. **Windowing** (`window.py`) — turns `(X, y)` into overlapping sliding windows of `WINDOW_SIZE` (`config.py`, default 20) past steps. `create_sliding_window()` produces one `(window, single-horizon-target)` pair per direct-forecast model call; `create_seq2seq_window()` produces one `(window, all-horizons-target)` pair for the seq2seq model. Random Forest additionally flattens the 3D window via `reshape_for_random_forest()` since it has no notion of sequence.
6. **Split** (`dataset.py`) — `time_series_split()` is a chronological (non-shuffled) train/test split — always split by index, never `train_test_split` with shuffling, since this is time-series data.
7. **Train/evaluate** (`experiments.py`) — `run_random_forest_experiment()`, `run_lstm_experiment()`, `run_seq2seq_experiment()` all follow the same shape: iterate ships × horizons, check for cached model+predictions on disk, otherwise window → split → train → predict → cache → evaluate (`evalute.py`: MAE/RMSE/R2) → plot (`visualization.py`). Results always end up as a DataFrame with columns `[ship, horizon, MAE, RMSE, R2]`, written to `eda_output/*_results.csv`, so RF/LSTM/seq2seq results are directly comparable.

## Model conventions (`src/models/`)

`RandomForestModel`, `LSTMModel`, and `Seq2SeqLSTMModel` all expose the same `train`/`predict`/`save`/`load` interface, letting `experiments.py` treat them interchangeably.

For the two LSTM models specifically:
- Both X and y are standardized (`StandardScaler`), fit **only on the training split** (never val/test) — predictions are inverse-transformed back to raw fuel units before evaluation, so metrics stay comparable across model types.
- Seq2seq's target scaler is fit jointly across all horizon columns (not one scaler per horizon), so the decoder learns one consistent output scale.
- Loss is `nn.HuberLoss` (not MSE) — several ships have positively-skewed targets with occasional high-consumption spikes; Huber caps the gradient impact of those outliers.
- Early stopping uses a **chronological** validation split (last `val_ratio` fraction of the train set, no shuffling) and restores the best-val-loss weights, not the final epoch's.
- Gradient clipping (`max_norm=1.0`) is applied every step.
- `hidden_size`/`num_layers`/`patience`/`loss_delta` are tunable per-ship via `per_ship_params` in `run_lstm_experiment()`/`run_seq2seq_experiment()` (see `main_lstm.py`/`main_seq2seq_lstm.py`) — Triton and Ceto are much smaller/noisier datasets than Poseidon and need smaller models + more patience to avoid overfitting; tune by ship, don't force shared hyperparameters.
- Model checkpoints (`.pt` via `torch.save`) bundle the fitted scalers alongside `state_dict`, since `predict()` needs the exact fit-time scaler to inverse-transform correctly.

## Known inconsistency

`dashboard_app.py` loads Random Forest models from `eda_output/rf_plots/` because `main.py`'s call to `run_random_forest_experiment()` passes `model_dir=RF_PLOT_DIR` instead of the `RF_MODEL_DIR` variable it defines but never uses (see comment in `dashboard_app.py`). If you fix `main.py` to use `RF_MODEL_DIR`, update `dashboard_app.py`'s `MODEL_DIR` (and retrain/re-save models) to match.

## Power-target investigation: persistence-bias in direct-forecast models (Poseidon)

Ongoing work under the power-target direction (see "Model conventions" above for `*_power.py` scripts). Starting point: TCN had the best MAE/RMSE of the 5 model families on Poseidon, but RF feature importance showed `Power_Lag1_t-1` at ~0.80 importance — i.e. models were mostly doing persistence forecasting ("predict the last known value"), which manifests as phase lag against the real trajectory, not genuine dynamics learning. This is not target-specific: `add_target_history_features()` (`preprocessing.py`) generates a `{target}_Lag1` feature from whatever `TARGET` is (was `Fuel_Lag1` before the fuel→power target switch, is `Power_Lag1` now) — the same trap recurs for any smooth/autocorrelated physical signal, regardless of unit.

- **DTW metric** (`evalute.py:dtw_distance()`): Sakoe-Chiba banded DTW (`window` param, default 10) with rolling 2-row storage — O(n·window) time / O(m) memory, since full O(n·m) DTW is infeasible at this project's test-set sizes (Poseidon's test split is ~21k points). Wired into `evaluate_regression()` → `experiments._scale_metrics()` → every experiment's `results_df`/CSV, alongside MAE/RMSE/R2. Quantifies how much error is recoverable by small time-realignment (persistence/phase-lag) vs. genuine magnitude error.
- **Checkpoint-selection experiments (ruled out)**: `TCNModel` supports `early_stop_metric="dtw"`/`"combined"` (select best checkpoint by validation DTW, or a weighted MAE+DTW blend, instead of validation Huber loss) — see `main_tcn_poseidon_dtw.py` / `main_tcn_poseidon_combined.py`. Result: negligible effect (<1% change, sometimes traded away RMSE for DTW). Conclusion: changing which already-trained epoch gets kept doesn't change what gradient descent optimizes for — the fix has to change the training target itself.
- **Delta-target fix (validated)**: `window.create_sliding_window_delta()` / `create_seq2seq_window_delta()` target `y(now+horizon) - y(now)` instead of the raw future value (`now` = last row of the window, so no leakage). Removes Lag1 as a free lunch — copying it now predicts delta≈0, wrong whenever the signal actually changes. Raw-unit predictions are reconstructed as `anchor + delta_hat` before evaluation, keeping metrics comparable to the raw-target baseline. Validated for TCN across all 4 horizons on Poseidon: MAE -16 to -47%, DTW -24 to -90%, R2 up at every horizon, with the biggest gains at short horizons (Lag1 is a better "cheat" at h=1 than h=20) — see `main_tcn_poseidon_delta.py` and its results CSV/plots (`eda_output_power/tcn_poseidon_delta_*`).
- **NaN training bug (fixed)**: delta targets sit at ~0 for long flat stretches, so many parameters see persistently tiny gradients; Adam's default `eps=1e-8` lets `sqrt(v_hat)+eps` be dominated by `eps`, and the next batch with real signal produces a blown-up update → NaN (compounded by `weight_norm`'s division by `||v||` in TCN specifically). Fixed by setting `adam_eps=1e-4` (verified empirically: eps=1e-8 corrupted ~12% of batches, eps=1e-4 corrupted zero) plus a defense-in-depth skip of any batch whose clipped gradient norm is non-finite. Applied to `TCNModel`, `LSTMModel`, and `Seq2SeqLSTMModel` (all three use plain Adam with no `eps` override otherwise).
- **Delta-target scripts also exist for RF/XGBoost/LSTM/Seq2Seq** on Poseidon (`main_rf_poseidon_delta.py`, `main_xgb_poseidon_delta.py`, `main_lstm_poseidon_delta.py`, `main_seq2seq_poseidon_delta.py`), written to check whether TCN is still the best architecture once every model gets the same fix. All write to their own `*_poseidon_delta*` result/model/prediction paths so the raw-target baselines stay untouched. **Current direction (as of 2026-07-21) deprioritizes this broadening** — the user wants to focus on Poseidon specifically and optimize the single best solution (TCN/Seq2Seq delta-target) further, not spread effort across more model families. LSTM-delta is explicitly deprioritized (LSTM underperforms Seq2Seq/TCN at longer horizons in the raw-target baseline). Not rolled out to Triton/Ceto or folded into the shared `experiments.py` pipeline (those all still target raw values).
- **Seq2Seq-delta ran, TCN-delta confirmed still the better architecture (2026-07-21)**: `main_seq2seq_poseidon_delta.py` (default hyperparameters, matching `SEQ2SEQ_PARAMS_BY_SHIP`'s Poseidon entry — `hidden_size=64, num_layers=1, dropout=0.3, weight_decay=1e-4`) was run on Colab; results in `eda_output_power/seq2seq_poseidon_delta_results.csv`. TCN-delta beats it at every horizon on every metric (MAE ~10-18% lower, DTW 2-3x lower at h1/h5) — the delta-target fix didn't just close the gap created by persistence bias, TCN genuinely generalizes better once that gap is removed. Conclusion: stop investing in Seq2Seq-delta, keep optimizing TCN-delta.
- **Optuna hyperparameter search added for TCN-delta (`main_tcn_poseidon_optuna.py`, 2026-07-21)**: one Optuna study per horizon (default `N_TRIALS=25`), searching `depth` (3-6), `width` (16/32/64/128, used as `num_channels=[width]*depth`), `kernel_size` (2/3/5/7), `dropout`, `learning_rate`, `weight_decay`, `batch_size`. Trials train with a smaller budget (`SEARCH_EPOCHS=60`, `SEARCH_PATIENCE=6`) than the final retrain of the winning config (`FINAL_EPOCHS=150`, `FINAL_PATIENCE=10`, i.e. `TCNModel`'s own defaults). Objective is deliberately **validation-set MAE in raw ShaftPower units** (anchor + delta_hat reconstruction on the same chronological val slice `TCNModel.train()` carves out internally via `VAL_RATIO=0.1`), not the internal scaled Huber `val_loss` — that loss lives on an arbitrary delta-scaler scale and isn't the metric anyone compares across configs. Optimizer stays Adam (not SGD) everywhere in this project; the delta-target NaN fix (`adam_eps=1e-4`) is Adam-specific and hasn't been re-verified for SGD, so switching optimizers isn't on the table without redoing that stability work.
  - **First full run's results, per horizon, final-retrain-on-test vs. the default-hyperparameter TCN-delta baseline**: h1 — tie (no meaningful change either direction). h5 — mixed: MAE/RMSE/R2 slightly worse, DTW ~32% better (trades magnitude accuracy for phase-alignment). h10 — clear win on every metric (MAE -10.6%, RMSE -0.8%, R2 +0.1%, DTW -38%). h20 — **worse on every metric** than default.
  - **Root-cause hypothesis for the h20 regression (not yet independently verified)**: GPU training of `TCNModel` (Conv1d + `weight_norm`) is likely not fully deterministic even with the fixed `RANDOM_STATE` seed — cuDNN conv ops can produce different floating-point results across runs unless `torch.use_deterministic_algorithms(True)` is explicitly set (not currently done anywhere in this project). Evidence: h20's winning trial (`depth=5, width=128, kernel_size=5, learning_rate=0.0006...`) reported val MAE ≈1.78MW during the search phase, but the final retrain with the *identical* hyperparameters got its best (lowest) val Huber loss at epoch 1 and never improved again, effectively landing on a barely-trained checkpoint — a very different outcome from what the search run must have produced to reach that val MAE. This means an Optuna "best" config found via one non-deterministic training run is not guaranteed to reproduce that quality on retrain, and by extension the h10 "win" (best params used `learning_rate=0.0034`, ~7x the architecture's default 5e-4 — visibly jerkier/non-monotonic train_loss trajectory, e.g. an outsized drop at epoch 13→14) hasn't been confirmed robust either, even though early stopping correctly locked onto epoch 9's checkpoint (lowest val_loss) regardless of what the trajectory did afterward.
  - **Next step (not yet done)**: re-run the h10 winning config 2-3 more times independently to check whether MAE stays stable around ~1.05MW or varies a lot run-to-run, before deciding whether to actually adopt it over the default hyperparameters. Only adopt a per-horizon Optuna config if the win is both large and reproducible; h1/h20 should keep default hyperparameters regardless (tie or regression). Best params per horizon are in `eda_output_power/tcn_poseidon_optuna_best_params.csv` (Colab Drive only — `eda_output_power/` is gitignored, this file doesn't exist in the git repo).
  - **Non-determinism hypothesis CONFIRMED (2026-07-22)**: a second full independent run of `main_tcn_poseidon_optuna.py` (identical code, identical search space) gave contradictory results vs. the first run. h1/h5 stayed roughly consistent (tie vs. default either way). h10 swung from "clear win" (-10.6% MAE) in run 1 to "basically tied" (-1.1% MAE) in run 2. h20 **flipped sign entirely**: run 1 was worse than default (+1.1% MAE), run 2 was clearly better (-9.2% MAE). Two runs of the same search picking different "best" hyperparameters and getting different final-retrain quality out of them confirms the search itself isn't trustworthy yet — no per-horizon Optuna config should be adopted from either run.
  - **Fix applied (`models/tcn.py`, commit `229abde`)**: `TCNModel.__init__` now sets `torch.backends.cudnn.deterministic=True`, `torch.backends.cudnn.benchmark=False`, and `torch.use_deterministic_algorithms(True, warn_only=True)` (plus `CUBLAS_WORKSPACE_CONFIG=:4096:8` set at module import time, required for deterministic cuBLAS/matmul on CUDA). Root cause: cuDNN's convolution algorithm selection (and its autotuner, `benchmark=True`, which was implicitly on by PyTorch's default) is not reproducible run-to-run on GPU even with a fixed seed — this is the standard, well-known source of this exact symptom, though it hasn't been re-verified with a third GPU run post-fix (only smoke-tested on CPU, where it doesn't crash but also doesn't exercise the cuDNN code path).
  - **Noise-robust search rework (`main_tcn_poseidon_optuna.py`, 2026-07-23)**: rather than re-running the same single-seed-per-trial search again to see if the cuDNN fix alone was enough, the search itself was made robust to whatever noise remains. `TCNModel` now accepts an optional `seed` (defaults to `RANDOM_STATE`, so every other caller is unaffected). Each Optuna trial trains `N_SEEDS=3` models (seeds `RANDOM_STATE..RANDOM_STATE+2`) per config and the objective returns the **mean val MAE across seeds**, not a single run's; `N_TRIALS` dropped 25→8 to keep total search compute roughly unchanged (`N_TRIALS * N_SEEDS ≈` old `N_TRIALS * 1`). The winning config's final evaluation is likewise an `N_SEEDS`-model ensemble (mean of raw-unit test predictions), not a single retrain — a single retrain reproducing a search run's quality is exactly what silently failed before (h20's winning trial: ~1.78MW val MAE during search vs. a near-untrained checkpoint on retrain). Per-seed MAE and their std are logged and written to `tcn_poseidon_optuna_best_params.csv` so a "winning" config that's actually high-variance across seeds is visible, not hidden behind one mean number. Smoke-tested locally (CPU, tiny data slice, `N_TRIALS=2`/`N_SEEDS=2`/2 epochs) end-to-end without errors; not yet run on Colab GPU. **Next actual step: run this reworked script on Colab and check whether h10/h20 now agree with themselves (via the logged per-seed std) before adopting any per-horizon config.**

## Caching

`models_saved/`, `predictions_cache/`, and `data_clean/` act as on-disk caches keyed by `{ship}_h{horizon}` (or `{ship}_seq2seq` for the seq2seq model). `use_cache=True` skips training and reloads from disk if both the model and predictions files exist. LSTM/seq2seq experiments default `use_cache=False` since hyperparameters change frequently during experimentation; Random Forest defaults `use_cache=True`.
